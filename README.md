## Общее описание проекта

**metricgen** — распределённая система для генерации синтетических метрик в формате Prometheus TSDB, предназначенная для нагрузочного тестирования, симуляции инфраструктурных сценариев и проверки масштабируемости хранилищ временных рядов (Thanos, VictoriaMetrics, Mimir и др.). Проект включает оркестратор, пул воркеров, очереди задач (Kafka / in‑memory), хранилище в S3 (блоки, схемы, метаданные задач и тренировочные данные), Web UI с дашбордом и калькулятором, а также полную наблюдаемость через Prometheus-метрики.

---

## Назначение системы

| Ключевая задача | Решение |
|----------------|---------|
| Генерация синтетических данных для Prometheus/Thanos | TSDB-блоки, совместимые с Thanos Store Gateway |
| Масштабирование до тысяч хостов и миллионов серий | Горизонтальное масштабирование воркеров через Kafka + KEDA |
| Управление заданиями через веб-интерфейс | SPA на Bootstrap + Alpine.js с real-time обновлениями (SSE) |
| Воспроизводимость и отказоустойчивость | Детерминированные генераторы, ретраи, DLQ, контрольный канал |
| Мониторинг и наблюдаемость | Prometheus-метрики, логи в UI, heartbeat воркеров |
| Долговременное хранение состояния | Все метаданные задач, схемы и тренировочные данные хранятся в S3, что обеспечивает восстановление оркестратора после рестарта |

## Ключевые компоненты и их реализация

### 1. Оркестратор

**Назначение**: центральный компонент, управляющий жизненным циклом заданий.

**Реализовано**:
- Приём заданий через REST API (`/api/generate`, `/submit`)
- Планирование: разбивка на окна (2 часа) и шарды по хешу серий
- Claim-check: схема сохраняется в S3, в задаче передаётся ссылка
- Публикация задач в очередь (Kafka или in-memory)
- Трекер прогресса (`JobTracker`) с агрегацией outcomes и SSE-пушами
- Хранение истории выполнения в S3 (`JobMetaStore`) — метаданные задач сохраняются в объекты `jobs_meta/<job_id>.json`
- Восстановление состояния трекера при старте оркестратора: загрузка всех `JobMeta` из S3
- Удаление задачи (`DELETE /api/jobs/{id}`) удаляет все данные: блоки по префиксу (`S3Prefix`), схему (`schemas/<job_id>.json`) и метаданные (`jobs_meta/<job_id>.json`)
- Пауза и возобновление заданий через control-канал (сохранение состояния паузы в трекере и синхронизация с воркерами)
- Мягкое удаление завершённых задач (`POST /api/jobs/clear-finished`)

### 2. Воркер (исполнитель)

**Назначение**: обработка отдельных задач — генерация TSDB-блоков.

**Реализовано**:
- Потребление задач из очереди (через `TaskQueue`)
- Подгрузка схемы из S3 (claim-check) с кэшированием по `job_id`
- Генерация серий (развёртывание топологии, декартово произведение)
- Детерминированные генераторы значений (синус, randomwalk, counter, uniform, constant)
- Каскадные отказы (downtime: хост → под → контейнер) с gap-сэмплами
- Запись TSDB-блока с корректным `meta.json` (совместим с Thanos)
- Загрузка в S3 с правильным порядком (chunks → index → meta.json)
- Публикация `TaskOutcome` в топик результатов
- Heartbeat (периодическая отправка статуса в control-канал)
- Реагирование на паузу (requeue без увеличения attempts) и отмену (дренирование) через control-канал

### 3. Очередь задач

**Назначение**: связь между оркестратором и воркерами, обеспечение отказоустойчивости.

**Реализовано**:
- Абстракция `TaskQueue` с методами `Publish`, `Consume`, `Stats`, `Close`
- Две реализации:
  - **In-memory** – для all‑in‑one режима и тестов
  - **Kafka** – для продакшена (персистентность, репликация, consumer groups)
- Ack/Nack с ретраями и DLQ (dead‑letter queue)
- Партицирование по `task_id` для равномерной нагрузки
- Контрольный канал для отмены, паузы и heartbeat
- Поддержка `NoPenaltyNacker` для паузы (requeue без увеличения попыток)

### 4. Хранилище (S3)

**Назначение**: долговременное хранение данных (блоки + схемы + метаданные + тренировочные данные).

**Реализовано**:
- Интерфейс `BlockStore` (Upload, Exists, DeletePrefix, ListSize, Close)
- Реализация на `thanos-io/objstore` (S3-совместимое) и `MemStore` для тестов
- Загрузка блоков с правильным порядком (сначала chunks/index, потом meta.json)
- Claim-check: схемы хранятся в папке `schemas/` как JSON
- Метаданные задач хранятся в `jobs_meta/<job_id>.json` (JobMetaStore)
- Тренировочные данные для ML — в `training/dataset.jsonl` (JSON Lines)
- Поддержка префиксов для изоляции разных источников
- Удаление всех объектов по префиксу (включая версии) через MinIO API

### 5. Генераторы значений

**Назначение**: создание реалистичных временных рядов.

**Реализовано**:
- Интерфейс `ValueGenerator` с методом `Value(timestampMs, seriesSeed)`
- Реализации:
  - `Sine` – синусоида (база + амплитуда × sin)
  - `Uniform` – равномерный шум [min, max]
  - `Counter` – монотонный счётчик с джиттером
  - `Constant` – постоянное значение (для info-метрик)
  - `PseudoWalk` – шум вокруг baseline
  - `RandomWalk` – stateful случайное блуждание
  - `Composite` – суперпозиция нескольких генераторов
- Фабрика `registry.go` для создания по имени (расширяемость)
- Детерминизм: генераторы возвращают одно и то же значение для одинаковых (seed, timestamp)

### 6. Топология и серии

**Назначение**: построение всех временных рядов из JobSchema.

**Реализовано**:
- Развёртывание топологии: хосты → поды → контейнеры
- Каскадные отказы: если хост упал, все его поды и контейнеры недоступны
- OwnDimensions: собственные лейблы метрики (cpu, mode, device)
- Декартово произведение для раскрытия лейблов
- Шардирование по хешу seed серии
- Расширение типов: summary → quantile-серии + _sum + _count

### 7. Веб-интерфейс

**Назначение**: управление генерацией и мониторинг через браузер.

**Реализовано**:
- SPA на Bootstrap 5 + Alpine.js (без сборок, embed в бинарник)
- Авторизация через cookie-сессии (логин/пароль)
- Дашборд с прогрессом джобов (плитки, прогресс-бары, статусы)
- Калькулятор масштаба (оценка числа серий, окон, объёма)
- Предгенерация схемы (построение JobSchema из параметров) с фоновым прогрессом и отменой
- Реальное время через SSE (события job, log, stats, pregen)
- Логи в кольцевом буфере (последние N записей)
- Мониторинг воркеров (heartbeat, количество обработанных задач)
- Управление заданиями (отмена, пауза, возобновление, удаление, повтор, очистка завершённых)
- Просмотр фактического размера данных задачи в S3 (GET /api/jobs/{id}/size)
- Тёмная тема

### 8. История и обучение модели

**Назначение**: накопление данных о выполненных задачах и прогноз объёма.

**Реализовано**:
- Хранилище метаданных задач в S3 (JobMetaStore)
- Восстановление состояния при старте оркестратора из S3
- Структуры: `JobMeta` (задание) и `TrainingEntry` (для ML)
- Сохранение параметров задания, ссылки на схему, фактического размера (`UploadBytes`)
- API для просмотра истории (`/api/jobs/history` — через `JobsHandler`)
- Экспорт данных для обучения в JSON Lines (dataset.jsonl) через `AppendTrainingData`
- (Планируется) ML-модель прогноза объёма на основе LightGBM (данные уже накапливаются)

## Инструкция по быстрому старту

### 1. Установка и сборка

```bash
git clone <repo>
cd metricgen
go mod tidy
go build ./...
```

### 2. Настройка окружения (переменные)

Пример `.env` для **all‑in‑one** режима:

```bash
MG_QUEUE_KIND=memory
MG_EMBEDDED_WORKERS=true
MG_HTTP_ADDR=:8080
MG_S3_ENDPOINT=10.5.7.33:30090
MG_S3_BUCKET=synthetic-store
MG_S3_ACCESS_KEY=your_key
MG_S3_SECRET_KEY=your_secret
MG_S3_INSECURE=true
MG_AUTH_ENABLED=true
MG_AUTH_USER=admin
MG_AUTH_PASSWORD=admin
MG_SCHEMA_PREFIX=schemas
```

### 3. Запуск

**All‑in‑one** (для разработки):

```bash
go run ./cmd/orchestrator
```

**Распределённо** (Kafka):

```bash
# Терминал 1 – оркестратор
MG_EMBEDDED_WORKERS=false go run ./cmd/orchestrator

# Терминал 2 – воркер 1
go run ./cmd/worker

# Терминал 3 – воркер 2 (опционально)
go run ./cmd/worker
```

### 4. Генерация схемы из реального /metrics

```bash
go run ./cmd/metrics -url http://node-exporter:9100/metrics \
  -max-dim-values=3 -max-dim-card=32 -drop-multilabel \
  -out configs/jobspec.generated.yaml
```

### 5. Отправка задания (CLI)

```bash
go run ./cmd/submit -spec configs/jobspec.generated.yaml \
  -from-hours-ago 2 -hours 2 -step-ms 15000 -series-per-shard 2000
```

### 6. Открыть веб-интерфейс

Откройте `http://localhost:8080`.  
Войдите с логином/паролем (по умолчанию `admin/admin`).

## Заключение

Проект **metricgen** представляет собой  систему для генерации синтетических метрик, готовую к развёртываниюи масштабированию в Kubernetes (манифесты пока отсутствуют, но могут быть легко добавлены). Он сочетает в себе:

- Высокую производительность (Go, горутины, конкурентность)
- Надёжность (ретраи, DLQ, heartbeat, graceful shutdown, персистентное состояние в S3)
- Управляемость (Web UI, REST API, CLI, мониторинг)
- Совместимость (Prometheus TSDB, Thanos, Grafana, Kafka, S3)

Архитектура построена на модульных интерфейсах, что позволяет легко адаптировать ядро для других задач (ETL, симуляция нагрузки, администрирование). Все ключевые решения задокументированы в ADR, а код покрыт юнит-тестами.
